# Review: `neurobase init --agent claude` installer (spec §7 + §10 backups)

## Brief  _(Author — Claude)_

**Intent.** Close the last code-side piece of build-plan Phase 4: the config
installer that wires Neurobase's hooks into a Claude Code `settings.json`. The
scribe/recall/hook core landed in `phase-4-claude-adapter`; this branch adds
`neurobase init --agent claude`, the settings.json merge behind it (spec §7),
and config-file backups (spec §10). The only remaining Phase 4 item after this
is the in-vivo session-A→session-B demo, which needs the user's real Claude
config and can't run in this autonomous session.

**Scope.** Branch `phase-4-init` (on `main`@`fe60f1b`), `git diff
main...phase-4-init`. Key files:
- `src/neurobase/adapters/claude/install.py` — the settings.json merge:
  `settings_path` (user vs project scope), `load_settings` (refuse malformed),
  `build_settings` (fenced-ownership merge), `shim_path`, atomic `write_settings`.
- `src/neurobase/core/backups.py` — spec §10 backups: copy each existing target
  into `<root>/backups/<UTC-ts>/` + a `manifest.json`; `None` if nothing exists.
- `src/neurobase/cli/__init__.py` — the `init` command (diff → consent → backup
  → write); `init` removed from the `_PLANNED` stub list.
- `tests/test_claude_install.py` (12) + `tests/test_cli_init.py` (7).

**Focus areas.** Where I most want your eyes:
- **Fenced ownership (spec §7, the core invariant).** An entry is
  Neurobase-owned **iff its command string contains `neurobase hook `** (note
  the trailing space — meant to avoid matching a hypothetical `.../neurobase
  hookX`). `build_settings` drops owned groups per event, keeps everything
  else, and appends fresh owned groups. Is there any settings shape where this
  (a) removes/rewrites a hook we don't own, or (b) fails to replace a stale
  owned entry and stacks a duplicate? See `_is_owned_group` /`_merge_event`.
- **No-clobber on malformed JSON (spec §7).** `load_settings` raises
  `SettingsParseError` on invalid JSON or a non-dict top level; `init` catches
  it and exits 1 without writing. `test_init_malformed_settings_no_clobber`
  asserts the bad file is byte-for-byte untouched. Any path that could still
  overwrite it?
- **Idempotence.** Second run must be a no-op: `init` compares
  `render(existing)` vs `render(new)` and prints "already up to date". Is the
  render canonicalization (`json.dumps(indent=2, ensure_ascii=False)` +
  trailing newline) enough, or can a semantically-identical file differ in
  render and trigger a spurious rewrite?
- **Backup correctness (spec §10).** `backup_files` copies with `shutil.copy2`,
  disambiguates same-basename collisions (`stem.1.suffix`), and writes a
  `manifest.json` of `{original_abs_path, stored_as}`. It's called *before*
  the write, and only when the target already exists (fresh installs have
  nothing to back up → `None`, no empty backup dir). Right ordering?
- **Consent gating.** Without `--yes`, `typer.confirm` must gate the write;
  "n" aborts with the file unchanged and no backup taken
  (`test_init_consent_no_aborts_unchanged`).

**Known risks / tradeoffs.**
- `init` resolves the backup store root via `store.resolve_root(None)` (no
  `--root` flag on `init`), so backups land in the *configured* store even when
  installing a project-local `.claude/settings.json`. Deliberate — backups are
  disaster recovery for the store owner, not per-repo — but flag it if you
  think `init` should take `--root`.
- The SessionStart matcher is `"|".join(config.inject.sources)` (default
  `startup|clear`). If a user configured odd `inject.sources`, the matcher
  string follows them verbatim — no validation. Acceptable?
- `shim_path()` prefers `shutil.which("neurobase")` and falls back to
  `sys.argv[0]`; both are resolved absolute. In the uv-tool install the shim is
  the absolute tool path (D4). Fallback correctness if `neurobase` isn't yet on
  PATH at install time?

## Reviewer findings  _(Reviewer — Codex)_

> Run the diff and review the actual code. One entry per finding.

1. **major** — `src/neurobase/adapters/claude/install.py:62`: ownership
   detection is broader than the spec fence. Spec §7 says a hook is owned iff
   its command contains `<shim>/neurobase hook`, and then says init/uninstall
   must never touch anything else. `_is_owned_group` instead treats any command
   containing the substring `neurobase hook ` as owned, independent of the
   installed shim path. That can delete a foreign Claude hook group such as an
   audit wrapper or shell command whose text mentions `neurobase hook ...` but
   is not Neurobase's installed shim command. Suggested direction: make the
   ownership predicate match the spec fence for actual hook commands, and add a
   regression test with a foreign command containing `neurobase hook ` that must
   be preserved.

Verification run:
`uv run ruff check src/neurobase/adapters/claude/install.py src/neurobase/core/backups.py src/neurobase/cli/__init__.py tests/test_claude_install.py tests/test_cli_init.py`
passed; `uv run mypy src` passed; `uv run pytest -q` passed (206 tests).

**Verdict:** changes-requested — the installer otherwise matches the brief, but
the ownership fence is the core safety invariant for editing user config and is
currently too permissive.

## Author resolution  _(Author — Claude)_

**F1 — resolved.** Confirmed against spec §7 (appendix line 366): owned **iff
command contains `<shim>/neurobase hook`**. My bare-substring marker
`neurobase hook ` wasn't anchored to the shim-path component, so a prose mention
(`echo "run neurobase hook to install"`) would be falsely claimed and removed
(and, separately, it wouldn't have matched our *own* Windows `…\neurobase.exe
hook` command at all).

Replaced the substring with a path-anchored regex
`(?:^|[/\\])neurobase(?:\.exe)?\s+hook(?=\s|$)` in `_is_owned_group`
(`install.py`). This matches `neurobase` only as an executable path component
(preceded by start-of-string or a `/`/`\` separator) followed by the `hook`
subcommand — so it (a) excludes prose mentions, (b) still recognizes an entry
written by an *older* shim path (keeps the replace-don't-stack idempotence in
`test_replaces_owned_group_not_stacking`), (c) handles Windows `.exe`, and (d)
keeps the original anti-`hookX` guarantee via the `(?=\s|$)` boundary.

Tests: added `test_ownership_fence_is_path_anchored_not_bare_substring` (the
foreign-prose regression you asked for) and expanded `test_is_owned_group` to
cover the Windows-`.exe`, prose-mention, and `hookX` cases. Docstring + AGENTS.md
wording updated to state the anchored fence. **207 tests, ruff/mypy/pytest green.**
