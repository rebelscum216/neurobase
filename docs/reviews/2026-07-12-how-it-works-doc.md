# Review: "How Neurobase works" codebase tour + README refresh

## Brief  _(Author — Claude)_

**Intent.** Add `docs/how-it-works.md` — a comprehensive, module-by-module tour
of the entire codebase so a new reader (human or agent) can understand every part
of `src/neurobase/` and how the subsystems fit together — and refresh `README.md`,
whose "Status" and "How it works" sections were stale (they claimed MCP and the
recommender were "still planned" when Phases 7–8 have shipped).

This is a **documentation-only** change. No source code, tests, or behavior are
touched. The bar for this review is therefore **factual accuracy against the
code**, not runtime correctness.

**Scope.** Branch `docs-how-it-works`, `git diff main...HEAD -- README.md
docs/how-it-works.md`. Key files:
- `docs/how-it-works.md` (new, ~1820 lines) — Part I orientation (what Neurobase
  is, an architecture diagram + module map, the three data-flow loops, the
  on-disk layout) followed by Part II, a per-subsystem reference with one `###`
  subsection per source file, real function signatures, invariants, and spec/ADR
  citations.
- `README.md` (rewritten) — corrected status, updated "How it works" bullets, a
  real command surface (`recommend`, `mcp serve`, `seed`, `status --recommender`),
  and a link to the new doc.

**How this doc was produced (context you should be skeptical of).** The doc was
drafted by a fan-out of subagents that each read one subsystem, then verified
against source. **5 of the 12 subsystem sections did NOT get an independent
verify pass** (the verifier agents hit a session limit) — for those I
re-verified signatures/claims against source myself, but a second independent
read is exactly what this relay is for. Please weight your attention toward them:
- **Claude Code adapter** (`## Claude Code adapter …`)
- **Codex CLI adapter** (`## Codex CLI adapter …`)
- **The CLI** (`## The CLI — Typer app …`)
- **Recommender II — ranker, proposal store, emitters, metrics**
- **Test suite** (`## Test suite …`)

**Focus areas.** In priority order:
1. **Factual accuracy of signatures and behavior.** Spot-check `###` file
   subsections against the actual code: do the function signatures match? Does the
   described control flow / invariant / edge case match what the code does? Flag
   any hallucinated symbol, wrong signature, or invented behavior.
2. **Spec/ADR citations.** The doc cites `spec §N`, `Dn`, `ADR-000n` throughout.
   Sample a few and confirm the citation points at the right contract (a wrong
   citation is misleading even in a doc).
3. **The Part I narrative** (loops, architecture diagram, on-disk layout) — is the
   data flow described correctly and consistently with the per-module reference?
4. **README accuracy** — are the status claims and command examples all real
   (command names/flags exist in `cli/__init__.py`)?

**Known risks / tradeoffs.**
- The 5 unverified sections above are the highest-risk for a stray inaccuracy.
- The Test suite section was deliberately condensed from a per-assert dump to a
  per-file coverage map; the per-file test *counts* were generated by
  `grep -cE '^\s*def test_' ` and may differ slightly from pytest's collected
  count once parametrization expands (the doc says "~450 … before parametrization"
  intentionally).
- Prose long-lines: the Part II sections are not wrapped at ~80 cols (the repo
  convention for docs) because they were machine-generated; Part I is wrapped.
  Flag as a `nit` if it matters, but reflowing 200KB was judged not worth it.

**How to verify.**
```bash
git diff main...HEAD -- README.md docs/how-it-works.md
# Spot-check signatures, e.g.:
grep -nE "^(def |class )" src/neurobase/recommender/ranker.py
grep -nE "^(def |class )" src/neurobase/cli/__init__.py
# Confirm README command names exist:
grep -nE "@app.command|@recommend_app.command|@mcp_app.command" src/neurobase/cli/__init__.py
# Sanity-check internal links resolve (all relative paths):
grep -oE "\]\(([^)]+)\)" docs/how-it-works.md
```

  `docs/notes/2026-07-09-phase-8-recommender-plan.md` — not part of this change
  and deliberately left out of the commit.
- Prose style/voice preferences (unless something is outright wrong or
  misleading). Copyedits welcome as `nit`s, not blockers.
- Any request to change the actual code — this is docs-only. If the doc describes
  a code smell, note it, but the fix belongs in a separate change.
