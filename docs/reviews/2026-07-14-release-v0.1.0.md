## Reviewer findings  _(Reviewer — Codex)_

> Run the diff and review the actual code. One entry per finding.

- **major — `pyproject.toml:1` — The release sdist is not isolated from the
  repo-local uv cache and currently ships machine-derived cache files.** A
  clean `uv build --out-dir /tmp/nb-release-dist` produced an sdist containing
  `.uv-cache/.lock`, `CACHEDIR.TAG`, nested cache control files, and
  `.uv-cache/interpreter-v4/.../*.msgpack`. The directory is gitignored but
  Hatchling still included it, so sdist contents depend on whatever ignored
  cache happens to exist on the build machine; the interpreter blob is also
  exactly the kind of local metadata that should not be published. The wheel
  itself is clean, and `.env` was absent from both artifacts, but PyPI publishes
  the sdist too. Suggested direction: explicitly exclude `.uv-cache/` (and
  preferably other development caches) from the sdist/build configuration,
  then rebuild and inspect both archive file lists from the same checkout that
  will be tagged.
  - **resolution:** resolved. Root cause was that `.uv-cache/` was never
    actually in `.gitignore` (despite the directory name suggesting it
    should be) — Hatchling's default sdist exclusion is inferred from VCS
    ignore state, so an untracked-but-not-ignored directory got swept in.
    Fixed at both layers: added `.uv-cache/` to `.gitignore` (root cause)
    and an explicit `[tool.hatch.build.targets.sdist] exclude = [".uv-cache/"]`
    (defense-in-depth, so this can't silently regress if gitignore ever
    changes). Verified with a fresh `uv build --out-dir /tmp/...`: sdist
    tarball now contains zero `.uv-cache`/`CACHEDIR` entries.

- **minor — `docs/how-it-works.md:277` — The public module reference still
  identifies the package version as `0.1.0.dev0`.** The version-bump diff
  updates `pyproject.toml` and `uv.lock`, but the sdist itself contains this
  contradictory identity documentation. Suggested direction: update the
  package identity line to `0.1.0` as part of the release-version sync.
  - **resolution:** resolved. `docs/how-it-works.md:277` now says `0.1.0`.

- **minor — `CHANGELOG.md:8` — The `0.1.0` release date predates the actual
  release.** The entry says `2026-07-13`, while this final pre-tag review and
  intended publication are occurring on `2026-07-14`. For a not-yet-published
  version, Keep a Changelog's version heading should record the real release
  date. Suggested direction: set it to the date the tag/GitHub Release/PyPI
  publication will actually occur.
  - **resolution:** resolved. `CHANGELOG.md`'s `[0.1.0]` heading now reads
    `2026-07-14`.

Verification performed: reviewed the full `main...HEAD` diff and README;
confirmed `.env` is ignored and untracked without reading it; built the exact
branch to `/tmp`; inspected wheel/sdist names, metadata, entry point, and every
archive path; confirmed wheel metadata `Name: neurobase-cli`, `Version: 0.1.0`,
`Requires-Python: >=3.11`; ran `git diff --check`; and reran the full gate
(ruff, format, mypy, `481 passed`). An isolated install attempted with the
system Python correctly refused because that interpreter is Python 3.9 and the
wheel requires 3.11+, so it was not treated as a package failure.

**Verdict: changes-requested** — version metadata and tests are correct, but
the sdist must not publish local cache state, and the two release-facing version
records should be synchronized before the irreversible PyPI upload.
