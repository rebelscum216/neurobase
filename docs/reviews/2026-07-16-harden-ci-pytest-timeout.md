# Review: add pytest-timeout guard so a hung test fails loud

## Brief  _(Author — Claude)_

**Intent.** Make the one CI gate (`scripts/ci.py` → `pytest`) fail-loud instead of
hang-forever. `make ci` hung mid-run and had to be Ctrl-C'd, leaving only a
`tmp_path` teardown `KeyError` with no test name. Root cause of *that* stall was
almost certainly a transient macOS system hiccup (the full suite passes reliably —
verbose, under a real pty, with a faulthandler watchdog — and could not be
reproduced), but the real gap it exposed is structural: `pytest-timeout` was never
installed, so any test that never returns (a subprocess that hangs, a blocking
stdin read) can stall the whole gate indefinitely with no indication of which test.
This change closes that gap.

**Scope.** Branch `harden-ci-pytest-timeout`, `git diff main...HEAD`. Key files:
- `pyproject.toml` — add `pytest-timeout>=2.3` to the `dev` dependency group; add
  `timeout = 60` under `[tool.pytest.ini_options]` with an explanatory comment.
- `uv.lock` — regenerated by `uv sync` (adds `pytest-timeout==2.4.0`).

**Focus areas.**
- Is `60` the right per-test ceiling? Suite runs in ~5s, so 60s is pure headroom;
  confirm it's high enough that no legitimate (slowest) test could false-trip, and
  low enough to be useful.
- Cross-OS correctness: the CI matrix runs macOS/Linux/Windows. Default method is
  `signal` (SIGALRM) on Unix — which actually interrupts a blocked syscall — and
  pytest-timeout auto-falls back to `thread` on Windows (no SIGALRM). Confirm no
  explicit `timeout_method` is needed and the fallback is genuinely automatic.
- Config placement: `timeout` as a dedicated ini key vs. folding into `addopts`.
  Chose the ini key so `--timeout=N` overrides still compose on the CLI.

**Known risks / tradeoffs.**
- New dev-group dependency. Dev-only (not in `[project.dependencies]`), so it never
  ships to end users or affects the wheel/sdist.
- A per-test timeout can mask a genuine "this test is legitimately slow" signal by
  converting it to a failure — mitigated by the large 60s headroom.

**How to verify.**
- `make ci` → green (801 passed in ~5s, ruff/format/mypy clean). Verified locally.
- Guard actually trips: a throwaway `def test(): time.sleep(30)` run with
  `--timeout=2` fails with `Failed: Timeout (>2.0s) from pytest-timeout` and
  `timeout method: signal` — confirmed locally (throwaway test not committed).

**Out of scope.**
- The transient hang itself — not reproducible and not believed to be a code bug;
  this change is the durable guardrail, not a fix for a specific reproduced defect.
- Any `src/` behavior — this touches dev tooling / test config only.
- The faulthandler-watchdog alternative (`-o faulthandler_timeout=…`) — considered
  and rejected in favor of per-test `pytest-timeout`; not part of this diff.
